# Tools Reference

VAIG provides tools that agents can call during their execution. Tools are organized by domain and registered via `ToolRegistry`.

## Tool Architecture

Each tool is defined as a `ToolDef`:

```python
@dataclass(frozen=True)
class ToolDef:
    name: str                    # Unique identifier
    description: str             # What the tool does
    parameters: list[ToolParam]  # Input parameters
    handler: Callable            # The function to call

@dataclass(frozen=True)
class ToolParam:
    name: str
    type: str                    # "string", "integer", "boolean", "array", "object"
    description: str
    required: bool = True
    enum: list[str] | None = None
```

Tools return a `ToolResult`:

```python
@dataclass
class ToolResult:
    success: bool
    output: str
    error: str | None = None
```

## File Tools

Available in **coding agent** mode (`--code` or `/code`). All file operations are sandboxed to the workspace directory.

### `read_file`

Read the contents of a file.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `path` | string | Yes | Relative path from workspace root |

**Returns:** File content as text.

**💡 When to use:** First step before any edit; reading config files; reviewing generated output.

```
read_file(path="src/main.py")
```

---

### `write_file`

Create or overwrite a file. Creates parent directories if needed.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `path` | string | Yes | Relative path from workspace root |
| `content` | string | Yes | File contents to write |

**Returns:** Confirmation with path and bytes written.

**💡 When to use:** Creating new files from scratch. Prefer `edit_file` or `patch_file` for modifications to existing files.

```
write_file(path="src/new_module.py", content="def hello(): ...")
```

---

### `edit_file`

Apply a targeted search-and-replace edit to an existing file.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `path` | string | Yes | Relative path from workspace root |
| `old_string` | string | Yes | Exact text to find (must match exactly once) |
| `new_string` | string | Yes | Replacement text |

**Returns:** Confirmation of the edit applied.

> **Note:** `old_string` must match exactly (including whitespace) and must appear exactly once. If not found or found multiple times, the edit fails.

**💡 When to use:** Small, targeted edits where you know the exact text to replace. For multi-hunk changes, use `patch_file` instead.

```
edit_file(path="src/main.py", old_string="print('debug')", new_string="logger.debug('debug')")
```

---

### `patch_file`

Apply a unified diff patch to a file atomically. Preferred over `write_file` for edits to existing files.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `path` | string | Yes | Relative path from workspace root |
| `patch` | string | Yes | Unified diff string with `@@` context headers |
| `backup` | string | No | Save backup before patching: `'true'` or `'false'` (default: `'false'`) |

**Returns:**
- Success: `{"success": true, "path": "..."}`
- Failure: `{"success": false, "error": "...", "conflicts": [...]}`

**💡 When to use:** Multi-hunk changes across a file; changes generated by diff tools; when you want atomic all-or-nothing application with conflict detection.

```
patch_file(
  path="src/config.py",
  patch="--- a/src/config.py\n+++ b/src/config.py\n@@ -10,1 +10,1 @@\n-DEBUG = True\n+DEBUG = False\n",
  backup="true"
)
```

### `list_files`

List files in a directory.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `path` | string | No | Directory to list (default: workspace root) |

**Returns:** Sorted list of entries (directories end with `/`). Skips hidden files and common ignored names (`.git`, `node_modules`, `__pycache__`).

**💡 When to use:** Discovering what files exist before reading; scoping searches.

```
list_files()
list_files(path="src")
```

---

### `search_files`

Search for a regex pattern across files in the workspace.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `pattern` | string | Yes | Regex pattern to search for in file contents |
| `path` | string | No | Directory to search (default: workspace root) |

**Returns:** Matching lines with file path and line number.

**💡 When to use:** Finding where a function is called; locating config values; cross-file symbol search.

```
search_files(pattern="def authenticate")
search_files(pattern="TODO", path="src")
```

---

### `verify_completeness`

Scan files for incomplete placeholder patterns. Checks for: `TODO`, `FIXME`, `HACK`, `XXX`, bare `pass` statements, ellipsis bodies (`...`), and `raise NotImplementedError`. Files larger than 1 MB are skipped with a warning.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `paths` | string | Yes | Comma-separated relative file paths to scan (e.g. `src/foo.py,src/bar.py`) |

**Returns:** Human-readable report in `file:line: pattern — 'matched text'` format, or "No incomplete patterns found" when all files are clean. `error` is only set on path safety or read errors — placeholder findings do **not** set `error=True`.

**💡 When to use:** Final verification step before submitting code; used by the Verifier agent in `CodingSkillOrchestrator` to confirm implementation completeness.

```
verify_completeness(paths="src/auth.py,tests/test_auth.py")
verify_completeness(paths="src/server.go")
```

## Shell Tool

### `run_command`

Execute a shell command in the workspace directory.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `command` | string | Yes | The command to run |

**Returns:** Combined stdout/stderr output (capped at 100,000 characters).

**Security constraints:**
- **Allowlist**: Only commands in `coding.allowed_commands` can be run (configurable)
- **Blocked patterns**: Dangerous argument patterns are always rejected (e.g., `rm -rf /`)
- **Timeout**: 30-second execution limit
- **Output cap**: 100,000 character maximum output
- **Confirmation**: Requires user confirmation if `coding.confirm_actions` is `true`

Default allowed commands include: `ls`, `cat`, `grep`, `find`, `wc`, `head`, `tail`, `sort`, `uniq`, `diff`, `python`, `pip`, `node`, `npm`, `git`, `make`, `cargo`, `go`, etc.

**💡 When to use:** Running tests, linters, builds, or any shell task that isn't covered by a dedicated tool.

```
run_command(command="python -m pytest tests/ -v")
run_command(command="git diff HEAD~1")
run_command(command="go build ./...")
```

## GKE Tools

Available in **live** mode (`vaig live` or `--live`). Require the `[live]` extra and a configured GKE cluster.

VAIG provides **23 base GKE tools** + **4 Helm tools** (enabled by default) + **5 ArgoCD tools** (opt-in) + **2 GCloud tools** — up to **34 infrastructure tools** total. Additionally, 5 file tools and 1 shell tool are available in coding agent mode.

### Read Operations

#### `kubectl_get`

List or get Kubernetes resources.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `resource` | string | Yes | Resource type (e.g., `pods`, `deployments`, `services`) |
| `name` | string | No | Specific resource name (omit to list all) |
| `namespace` | string | No | Namespace (default from config). Use `all` for all namespaces |
| `output_format` | string | No | Output format: `table` (default), `yaml`, `json`, `wide` |
| `label_selector` | string | No | Label filter (e.g., `app=nginx,tier=frontend`) |
| `field_selector` | string | No | Field filter (e.g., `status.phase=Running`) |

Supports 25 resource types with aliases: `pods`/`po`/`pod`, `deployments`/`deploy`, `services`/`svc`, `configmaps`/`cm`, `secrets`, `ingresses`/`ing`, `statefulsets`/`sts`, `daemonsets`/`ds`, `jobs`, `cronjobs`/`cj`, `namespaces`/`ns`, `nodes`, `persistentvolumeclaims`/`pvc`, `persistentvolumes`/`pv`, `serviceaccounts`/`sa`, `networkpolicies`/`netpol`, `horizontalpodautoscalers`/`hpa`, `replicasets`/`rs`, `endpoints`/`ep`, `poddisruptionbudgets`/`pdb`, `resourcequotas`/`quota`.

**Returns:** Formatted table or YAML/JSON depending on `output_format`.

**💡 When to use:** First step in any triage — see what's running, what's not, resource counts.

```
kubectl_get(resource="pods", namespace="production", label_selector="app=api")
kubectl_get(resource="deploy", namespace="all")
kubectl_get(resource="hpa", namespace="production", output_format="yaml")
```

#### `kubectl_logs`

Get container logs from a pod. Automatically fetches previous container logs when the current container is in `CrashLoopBackOff`.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `pod` | string | Yes | Pod name |
| `namespace` | string | No | Namespace |
| `container` | string | No | Container name (for multi-container pods) |

| `since` | string | No | Duration filter (e.g., `1h`, `30m`, `1h30m`) |

**Returns:** Log lines, with automatic fallback to previous container logs on CrashLoopBackOff.

**💡 When to use:** Diagnosing application errors, exceptions, startup failures, or crash reasons.

```
kubectl_logs(pod="api-server-xyz", namespace="production", tail_lines=200)
kubectl_logs(pod="api-server-xyz", container="sidecar", since="30m")
```

#### `get_events`

List Kubernetes events in a namespace, filtered by type and/or involved object. Events reveal WHY pods fail, WHY nodes have issues, and what the scheduler is doing.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `namespace` | string | No | Namespace (default from config) |
| `event_type` | string | No | Filter by type: `Warning`, `Normal`, or omit for all |
| `involved_object_name` | string | No | Filter by involved object name (e.g., a pod or node name) |
| `involved_object_kind` | string | No | Filter by involved object kind (e.g., `Pod`, `Node`, `Deployment`) |
| `limit` | integer | No | Max events to return (default: 50, max: 500) |

**Returns:** Events sorted by time with reason, message, count, and involved object.

**💡 When to use:** Critical for SRE triage — always check events when a pod isn't starting, a deployment is stuck, or nodes seem unhealthy.

```
get_events(namespace="production", event_type="Warning")
get_events(namespace="default", involved_object_name="api-server-xyz", involved_object_kind="Pod")
get_events(namespace="production", limit=100)
```

**Returns:** Rollout status including ready/desired replicas, conditions, and strategy type.

#### `get_rollout_history`

Show the revision history of a Kubernetes deployment. Lists all revisions by examining ReplicaSets owned by the deployment. When a specific revision number is provided, shows detailed pod template information for that revision.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | Yes | Deployment name |
| `namespace` | string | No | Namespace |
| `revision` | integer | No | Specific revision number for detailed view. Omit to list all revisions |

**Returns:** List of revisions with change cause, or detailed pod template (containers, images, resource limits) for a specific revision.

**💡 When to use:** Before recommending a rollback — understand what changed in each revision. Compare images between revisions.

#### `get_node_conditions`

Show node health conditions, resource pressure, taints, and capacity.

- Without a node name: lists ALL nodes with status, roles, version, OS, kernel, container runtime, and CPU/memory capacity.
- With a node name: shows detailed conditions (MemoryPressure, DiskPressure, PIDPressure, NetworkUnavailable), taints, labels, and capacity vs. allocatable.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | No | Specific node name for detailed view. Omit to list all nodes |

**Returns:** Node summary table or detailed per-node conditions.

**💡 When to use:** When pods are being evicted, failing to schedule, or nodes show abnormal behavior. Fills the gap where `kubectl_get nodes` only shows Ready/NotReady but hides pressure conditions.

```
get_node_conditions()
get_node_conditions(name="gke-prod-pool-abc123")
```

Reports: name, image, state (Waiting/Running/Terminated with reason and exit code), ready flag, restart count, last termination state, resource requests/limits, volume mounts, and env var source names (no secret values exposed).

Check whether a service account or the current user has permission to perform a specific action on a Kubernetes resource. Uses SubjectAccessReview (for service accounts) or SelfSubjectAccessReview (for current user).

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|

**💡 When to use:** Before operations that might fail with permission errors; auditing what a CI service account can do; debugging RBAC issues.

### Exec Operations

> **Security:** Exec operations are **disabled by default** and require explicit opt-in via `gke.exec_enabled: true`.

#### `exec_command`

Execute a diagnostic command inside a running container.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `pod_name` | string | Yes | Pod name |
| `namespace` | string | Yes | Namespace |
| `command` | string | Yes | Diagnostic command to execute (must match allowlist) |
| `container` | string | No | Container name (for multi-container pods) |
| `timeout` | integer | No | Execution timeout in seconds (default: 30, max: 300) |

**Returns:** Command output, truncated to 10,000 characters.

**Security model (three layers):**

1. **Config gate:** `gke.exec_enabled` must be `true` (disabled by default)
2. **Denylist:** Commands matching dangerous patterns are always rejected (shell metacharacters `; & | \``, redirection `>`, destructive commands like `rm`, `kill`, `sudo`, `chmod`, etc.)
3. **Allowlist:** Command must start with an allowed prefix: `cat`, `head`, `tail`, `ls`, `env`, `printenv`, `whoami`, `id`, `hostname`, `date`, `ps`, `top -bn1`, `df`, `du`, `mount`, `ip`, `ifconfig`, `netstat`, `ss`, `nslookup`, `dig`, `ping`, `curl`, `wget`, `nc`, `java -version`, `python --version`, `node --version`, `cat /etc/resolv.conf`, `cat /etc/hosts`

**💡 When to use:** Checking DNS resolution inside a pod; inspecting environment variables; validating connectivity from inside the network namespace. Always prefer read-only diagnostic commands.

```
exec_command(pod_name="api-server-xyz", namespace="production", command="cat /etc/resolv.conf")
exec_command(pod_name="api-server-xyz", namespace="production", command="ps aux", container="app")
exec_command(pod_name="debug-pod", namespace="default", command="curl -s http://localhost:8080/healthz", timeout=10)
```

### Labels & Annotations Tool

#### `kubectl_get_labels`

Get labels and annotations for Kubernetes resources. Supports filtering by label selector (server-side) or annotation prefix (client-side). Essential for detecting management context (GitOps, Helm, Operator).

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `resource_type` | string | Yes | Resource type (e.g. `deployments`, `pods`, `services`) |
| `namespace` | string | No | Namespace (default from config) |
| `name` | string | No | Specific resource name. Omit to list all |
| `label_filter` | string | No | Kubernetes label selector (e.g. `app.kubernetes.io/managed-by=Helm`) |
| `annotation_filter` | string | No | Annotation key prefix to match — client-side (e.g. `argocd.argoproj.io`) |

**Returns:** Table of resources with their labels and filtered annotations.

**💡 When to use:** Before writing operations — detect if a resource is Helm- or ArgoCD-managed to understand change impact. Finding all resources owned by a team or app.

```
kubectl_get_labels(resource_type="deployments", name="my-app", namespace="production")
kubectl_get_labels(resource_type="deployments", namespace="production", label_filter="app.kubernetes.io/managed-by=Helm")
kubectl_get_labels(resource_type="deployments", namespace="production", annotation_filter="argocd.argoproj.io")
```

### Discovery Tools

High-level cluster discovery tools that provide topology overviews. All support a `force_refresh` flag to bypass the in-memory cache.

#### `discover_workloads`

High-level overview of all workloads in a namespace. Unhealthy workloads are surfaced first.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `namespace` | string | No | Namespace (default from config) |
| `include_jobs` | boolean | No | Include Jobs and CronJobs (default: false) |
| `include_rollouts` | boolean | No | Include Argo Rollout resources (default: false) |
| `force_refresh` | boolean | No | Bypass cache (default: false) |

**Returns:** Summary of Deployments, StatefulSets, DaemonSets (and optionally Jobs/Rollouts) — name, replicas ready/desired, health status.

**💡 When to use:** Cluster-wide health check at the start of an investigation; getting a quick view of what's unhealthy without querying each resource type individually.

```
discover_workloads(namespace="production")
discover_workloads(namespace="production", include_jobs=true, include_rollouts=true)
```

#### `get_mesh_config`

Show active Istio traffic management config: VirtualServices, DestinationRules, and Gateways.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `namespace` | string | No | Namespace |
